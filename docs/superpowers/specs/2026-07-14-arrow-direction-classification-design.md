# 箭头方向二阶段识别设计

## 目标

在现有停车场目标检测基础上，对 `direction_arrow` 检测框进一步识别为直行、左转或右转，并将稳定方向与置信度返回 Android APP。当前阶段只展示和记录结果，不自动控制车辆。

## 架构

视觉服务继续使用现有停车场检测 ONNX 模型。检测到 `direction_arrow` 后，服务按检测框向外扩展 8% 并裁剪原始画面，将裁剪图送入箭头分类 ONNX 模型。

分类模型由 `model/arrow_cls_best.pt` 转换为 ONNX，在 Jetson 上通过 OpenCV DNN 执行。服务不得为分类功能加载 PyTorch，以保持当前低内存控制链路。

## 数据流

1. 从 8000 端口读取原始 MJPEG。
2. 原检测模型以当前频率检测停车场目标。
3. 对每个 `direction_arrow` 框执行方向分类。
4. 分类置信度低于 0.5 时保留通用箭头结果，不输出具体方向。
5. 最近 5 次有效分类采用至少 3 票的多数投票，产生稳定方向。
6. 8200 端口返回检测框、单帧方向、方向置信度和稳定方向。
7. APP 在视频框和检测列表显示“直行 / 左转 / 右转”。

## 接口

箭头检测项增加可选字段：

```json
{
  "label": "direction_arrow",
  "confidence": 0.91,
  "direction": "turn_left",
  "direction_confidence": 0.93,
  "stable_direction": "turn_left"
}
```

非箭头检测项及方向未确认时保持兼容，新增字段允许缺省。方向值限定为 `go_straight`、`turn_left`、`turn_right`。

## APP 展示

- 检测列表：`方向箭头 · 左转 · 93%`
- 视频叠加框：稳定方向优先；尚未稳定时显示本帧方向；置信度不足时仍显示“方向箭头”
- 方向采用独立颜色和中文名称，但不新增控制按钮

## 异常处理

- 分类模型缺失或加载失败：主检测继续运行，服务状态报告分类模型错误。
- 裁剪框无效：跳过该框分类。
- 分类结果未知：忽略具体方向，不影响其他目标。
- 无连续有效箭头：清空稳定投票，避免旧方向长期残留。

## 性能边界

- 分类只处理箭头裁剪区域，不处理整帧。
- OpenCV 线程保持 2，主检测上限保持 3 FPS。
- 开启分类后控制 HTTP 往返目标仍低于 20 ms。
- Jetson 视觉服务内存目标低于 300 MB，不允许重新引入多 GB PyTorch 常驻进程。

## 验证

- 单元测试覆盖裁剪、方向字段、置信度阈值、多帧投票和旧方向清理。
- 使用三类箭头样本验证模型转换前后分类一致。
- 小车实测三类箭头，检查 APP 文本与叠加框。
- 视觉运行时复测前进和停止请求延迟。
